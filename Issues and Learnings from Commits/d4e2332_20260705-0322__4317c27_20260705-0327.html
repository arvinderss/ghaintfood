Issue 1: Menu descriptions don't switch language

commit d4e2332

Reported

Descriptions stay in English after switching to hi/pa

Traced

Followed the render path from menu data → translate function → DOM

Root cause

Code read a field that was never defined in the data file

Fix applied

Added a translation lookup table + rewired the function

Verified

Headless-browser test confirms text switches in both languages

The problem

Ghaint Food's site has a working English → Hindi → Punjabi switcher. Category names ("Burgers" → "बर्गर"), item names ("Aloo Tikki" → "आलू टिक्की") and all static page copy translated correctly. But the long marketing description text under every menu item card stayed in English no matter which language was selected — the one piece of visible content that customers actually read to decide what to order.

Root cause

Every other translatable string in the codebase follows the same pattern: js/i18n.js holds a lookup object (MENU_I18N) keyed by the canonical English text, and a small translate*() helper in js/main.js looks up the current language in that object. Descriptions were the exception — the code for them was written to expect a per-item field called descriptionTranslations directly on each menu item object, but that field was never added to js/menu-data.js. It doesn't exist anywhere in the repository.

js/main.js — before (the bug)
function translateDescription(item) {
  if (currentLang === "en") return item.description || "";
  const translations = item.descriptionTranslations || {};  // ← always {} — field never set
  return translations[currentLang] || item.description || ""; // ← always falls back to English
}

Because item.descriptionTranslations was always undefined, the function silently fell back to the English string on every call — no error, no warning, just quietly wrong output.

Approach

  • Read js/menu-data.js (the menu content), js/i18n.js (translation tables) and js/main.js (render + language-switch logic) end to end to understand how translation actually flows through the app.
  • Confirmed the existing, working pattern used for categories/items/notes/tiffin — a lookup table in MENU_I18N keyed by the English string — and decided the fix should follow that same convention rather than inventing a new one.
  • Wrote Hindi and Punjabi translations for all 40 menu item descriptions across every category (Burgers, Pizzas, Loaded Fries & Sides, Garlic Bread, Subs & Sandwiches, Salads, Shakes, Lassi & Coolers, Combos) and added them as a new MENU_I18N.descriptions map.
  • Rewired translateDescription() to read from that map instead of the nonexistent per-item field.

The fix

js/main.js — after
function translateDescription(item) {
  if (!item.description) return "";
  return localize(MENU_I18N.descriptions, item.description);
}
js/i18n.js — added (excerpt of 40 entries)
MENU_I18N.descriptions = {
  "A crisp golden potato and pea tikki, packed with cumin and
   green chili...": {
    hi: "मसालेदार जीरा और हरी मिर्च से भरपूर, कुरकुरी सुनहरी...",
    pa: "ਜੀਰੇ ਤੇ ਹਰੀ ਮਿਰਚ ਨਾਲ ਭਰਪੂਰ, ਕੁਰਕੁਰੀ ਸੁਨਹਿਰੀ..."
  },
  /* … 39 more item descriptions, same shape … */
};

This mirrors the existing translateItem() / translateCategory() pattern exactly, so no other part of the render pipeline needed to change — renderMenu() already called translateDescription(item) per card.

Expected vs. actual outcome

Expected: selecting Hindi or Punjabi translates every visible menu string, including descriptions, matching the behaviour already working for names and categories.
Actual: confirmed via automated browser test — description text now switches scripts correctly for both languages, on the first menu item checked and by inspection of the full translation map (40/40 items covered). No regressions: English mode is untouched, and the cart/WhatsApp/UPI logic (which intentionally always uses the canonical English name internally) was not affected.

Issue 2: Undefined --ink-muted CSS variable

commit 4317c27

The problem

While reviewing the diff before committing the Issue 1 fix, an unrelated uncommitted change was found already sitting in the working tree in css/styles.css. The .menu-card-description rule — the exact element whose translation was just fixed — set its text color to var(--ink-muted). That custom property is never defined anywhere in the theme's :root block (light or dark), so the browser falls back to the inherited/default color instead of the intended muted tone used for secondary text everywhere else on the site.

Approach & verification

  • Searched the whole stylesheet for --ink-muted — zero definitions, zero other usages. Confirmed it was a typo/leftover, not an intentionally-undefined "inherit from parent" pattern used elsewhere in the file.
  • Checked what the surrounding rules use for the same "secondary/soft text" role — --ink-soft — defined in every theme variant (light, dark, and the explicit data-theme overrides) and used consistently for prices, notes, cart labels, etc.
  • Since this file wasn't part of the original task, it was flagged to the user first rather than folded in silently — then committed separately once confirmed it was wanted.
css/styles.css
.menu-card-description { color: var(--ink-muted); font-size: 14px; line-height: 1.5; margin-top: 4px; }
.menu-card-description { color: var(--ink-soft); font-size: 14px; line-height: 1.5; margin-top: 4px; }

Expected vs. actual outcome

Expected: menu card description text renders in the theme's intended muted ink tone, in both light and dark mode, matching every other piece of secondary text on the card (price label, "takeaway" tag).
Actual: one-line variable swap, verified by inspection — --ink-soft resolves correctly in all three theme contexts defined in the stylesheet, with no other rule depending on the removed --ink-muted name.

General learnings

  • Silent fallbacks hide bugs. Both issues share a shape: code that references something undefined (a JS object field, a CSS variable) doesn't throw — it just quietly degrades to a default. That's convenient for resilience but dangerous for correctness, since nothing surfaces the problem until a human happens to notice.
  • Consistency check as a diagnostic. Noticing that descriptions were the only field not following the established MENU_I18N lookup pattern was the fastest route to the root cause — inconsistent patterns in a codebase are worth treating as a signal, not just a style nit.
  • Verify in a real browser, not just by reading code. A script cross-checking translation-key coverage proves the data is complete; it doesn't prove the UI renders it. Actually loading the page, clicking the language switch, and reading the rendered DOM is what turns "should work" into "confirmed working."
  • Scope discipline. The CSS variable bug was found by accident while preparing to commit an unrelated fix. It was surfaced and committed separately, on its own merits, rather than bundled silently into the description-translation commit.
